# **Board Game Café Lounge \& Session Tracker**

### 

### **Description**



The Board Game Café Lounge \& Session Tracker is a console-based, headless REST API backend designed to optimize inventory tracking and table allocations for modern hospitality venues. Unlike traditional relational database systems that rely on rigid foreign key constraints and resource-heavy table joins, this system utilizes MongoDB to demonstrate the flexibility of NoSQL document references.



When a customer group arrives at the café, the backend coordinates a multi-collection transaction: it verifies table capacity, checks the availability of a requested board game, dynamically updates states across independent collections, and establishes an active session document. This architecture ensures high operational data integrity, preventing logical conflicts such as double-booking tables or simultaneous checkouts of a single physical asset. The entire system is built strictly as a backend server, relying on structured JSON payloads for communication.



### **Tools \& Technologies Used**



**Programming Language**: Python 3.10+

**Backend Framework**: FastAPI (Asynchronous, high-performance web routing framework)

**Database Engine**: MongoDB (Local Community Server instance)

**Database Driver**: PyMongo (Official Python driver for MongoDB integration)

**Database GUI**: MongoDB Compass (For real-time visual schema monitoring)

**Data Validation Engine**: Pydantic (Enforces data type serialization and request payload validation)

**API Client**: Postman (Utilized exclusively for endpoint testing, CRUD execution, and response evaluation)



### **Modules of the REST API**



The application architecture is structured around four primary entity modules mapping to independent collections in MongoDB, supported by an automated system utility tool.

##### **Module 1: Game Categories**

This module manages the core classification of the café’s entertainment inventory (e.g., Strategy, Party, cooperative games).

POST /categories/ (Create) — Ingests and stores a new game genre description.

GET /categories/ (Read) — Fetches all active classifications from the database.

DELETE /categories/{id} (Delete) — Removes a specific category using its unique Object ID.

DELETE /categories/drop/all (Drop) — Completely purges the categories collection for database maintenance.

##### 

##### **Module 2: Board Games Inventory**

This module maintains the status and structural attributes of individual board games. It exhibits a One-to-Many relationship, as multiple games map back to a singular category reference.



POST /games/ (Create) — Registers a game box into inventory after validating that its category\_id exists.

GET /games/ (Read) — Displays the complete inventory, dynamically stitching the matching category data onto each game payload.

DELETE /games/drop/all (Drop) — Flushes the entire games inventory collection.



##### **Module 3: Table Layout Management**

This module tracks the spatial assets of the café, handling seating capacity configurations and real-time availability states (is\_occupied: True/False).

POST /tables/ (Create) — Adds a new physical table to the venue layout map

GET /tables/ (Read) — Returns the real-time occupancy and capacity status of all tables.

DELETE /tables/drop/all (Drop) — Purges the table floor-plan layout collection.



##### **Module 4: Live Café Sessions (The Relational Mapping Hub)**

The core transactional component of the application. It creates a Many-to-One mapping matrix by consolidating a customer group, an assigned table, and a borrowed game into a single operational document.

POST /sessions/ (Create \& Update) — Accepts a Table ID and a Game ID. The module triggers cross-collection verification: if both assets are free, it modifies the table to Occupied, flags the game as Being Played, and commits the active session document.

GET /sessions/ (Read) — Aggregates and returns active sessions, outputting a nested JSON payload containing the group names along with their resolved table numbers and game titles.

DELETE /sessions/drop (Drop) — Drops the sessions collection, instantly resetting the venue's assets back to default states.

##### 

##### **System Utility Tool**

POST /setup-dataset (Database Seeder) — An automation route that clears old historical entries and instantly populates MongoDB Compass with a pre-configured mock dataset, ensuring immediate, out-of-the-box evaluation inside Postman.

### 

### **API Execution Lifecycle (Postman Workflow)**

To prove the robustness of the backend, the system follows a strict transaction logic flow when creating a live customer session:

**Client Request**: Postman sends a POST request to /sessions with a payload linking a table\_id and a borrowed\_game\_id.

**State Verification**: The backend queries the tables collection to ensure is\_occupied is false, and queries the games collection to ensure status is "On Shelf".

**Atomic State Update**: If verified, the backend executes two simultaneous updates:

Switches the table’s is\_occupied flag to true.

Switches the game’s status flag to "Being Played".

**Document Creation**: The session document is successfully created and committed to the database.

**Exception Handling**: If a user attempts to book the same table or game again, the backend intercepts the request and throws an HTTP 400 Bad Request error, preventing duplicate entries.
